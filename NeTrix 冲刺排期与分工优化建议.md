# NeTrix 冲刺排期与分工优化建议 (针对 4 周 MVP)

基于《NeTrix 技术栈与开发框架方案》中的排期与分工，整体架构非常专业且具备实操性，特别是“做与假装做（Build vs Fake）”的策略，是真正的黑客松/MVP 赢家思维。但在时间紧迫（4周）且包含期末复习压力的前提下，仍存在一些隐患。

以下是深度评估与优化建议，建议在技术例会上与团队同步：

## 一、 关于“人员分工”的致命隐患与调整

原方案逻辑（CS做前端，EEE做BaaS，Math做算法）看似完美，但忽略了一个现代全栈开发的现实：**在这个项目中，前端（Next.js）的工作量极其庞大。**

### 🚨 潜在隐患
1. **CS 同学（前端主力）会成为绝对的瓶颈：** 一个人要画赛博朋克 UI、跑通 React Flow 节点图、做论坛 CRUD 界面、做雷达动画，还要对接另外两人的 API。很容易在第二、第三周崩溃。
2. **EEE 同学（BaaS）后半程可能“没事干”：** Supabase 的建表、RLS 和 Edge Function 搭建非常快，熟练的话两三天就能配完。

### 💡 优化建议（必须调整）
1. **打破前后端次元壁：** 既然使用了 Next.js，**EEE 同学必须分担前端的数据绑定工作**。
   - **CS 同学**：专职负责“画出好看的静态组件（UI）”和 React Flow 核心画布的视觉交互动画。
   - **EEE 同学**：负责写数据获取代码（Supabase SDK Fetching），把数据库里的帖子动态渲染到 CS 同学画好的骨架里。
2. **算法同学的额外任务：** 匹配算法迁移其实非常快。Math 同学在第一周写完 API 后，必须负责**生成高质量的“假数据（Seed Data）”**。比如写一个 Python 脚本，通过大模型批量生成 50 个赛博朋克的假用户画像、20 篇带上下文的帖子，直接塞进数据库。这能极大加速前端的开发和最终演示的真实感。

## 二、 应对全员期末断崖式排期的“终极逃生指南”（针对 4.26 - 6.8）

既然整个团队（CS, EEE, Math）在 **5月11日 - 5月28日** 期间都将进入期末大闭关，这意味着原定 4 周连续开发的计划彻底作废，**团队真实的开发时间已被切割为两段总计不足 2.5 周的碎片时间**。

如果强行走“前端+微服务+BaaS”的架构，100% 会在联调阶段遭遇滑铁卢导致烂尾。必须执行**“全员重度 Fake，纯前端突破”**战略。

### 🛡️ 阶段一：全员基建与静态铺场（4.26 - 5.11，约 15 天）
既然没有后端可用，**团队所有人必须全部扑向 Next.js 前端体系！**
- **CS 同学（视觉核心）：** 彻底放弃真实数据联调。只负责用 Tailwind CSS 和 React Flow 搭建静态的、视觉炫酷的组件（登录壳子、节点地图、对话框、雷达扫描 CSS 动画）。
- **Math 同学（数据提供与 AI）：** 放弃将 FastAPI 部署到 Render（极其耗时）。用你写好的 Python 算法在本地电脑上跑出 50 份优秀的匹配结果，直接打包导出成一个庞大的 `fakeData.json` 给 CS 同学用。另外，研究怎么直接用前端 fetch 调用 ChatGPT API 来做“AI 召唤”。
- **EEE 同学（数据对接）：** 放弃 Supabase 复杂的 RLS 权限和多表关联。负责写一套最简单的“伪后端”代码，利用 `localStorage` 或 Next.js 的本地状态，把 Math 导出的由 JSON 数据“喂”进 CS 画好的界面里，让界面看起来能点动。

### ⏳ 阶段二：全员冻结期（5.11 - 5.28）
- **状态：** 项目在 GitHub 上静默。大家安心准备期末考试。此时的项目状态必须是一个“虽然数据都是假的，但是打开网页点一点不会报错白屏”的纯静态前端 Demo。

### 🚀 阶段三：强势回归与纯视觉收尾（5.28 - 6.8）
- **状态：** 考完回归，决战路演。
- **核心任务（绝不去碰后端的 Bug！）：** 
  1. **固化路演剧本：** 团队死磕那 3-5 分钟的现场演示 PPT。精确到鼠标先点哪里，再点哪里。
  2. **定制演示数据：** 根据路演剧本，把系统里预埋的假 JSON 数据修改为最有戏剧效果、最贴合“H2A2H”故事内容的数据（比如硬写入一段极具启发性的 AI 对答）。
  3. **视觉微调：** 修复因分辨率导致的 UI 错位，增加 React Flow 的霓虹发光动效，确保大屏幕上够炫。

## 三、 MVP 极限“瘦身”策略 (Demo-Driven Development)

在4周时间里强行追求“业务逻辑的完整性”是非常危险的。对于MVP而言，**凡是评委在路演的那 3-5 分钟里看不到或点不到的功能，都不值得花精力和时间开发。** 必须采用“好莱坞影视城”策略（正面宏伟，背面纸板）：

1. **认证系统 (Auth) — 从“真实登录”降级为“一键切号”：** 
   - **优化：** 砍掉冗长的注册、邮箱验证流程。做一个“角色选择台”打底页，比如放置大按钮“以【求知者】身份进入”、“以【先知】身份进入”。点击后前端强行写入本地缓存（Local Storage），直接跳转主页模拟登录状态。
   - **成效：** 至少节省2天开发和测试时间。

2. **知识中心 (Database) — 从“全量真实”降级为“单点真实”：**
   - **优化：** 不必做一套能够承载所有学科的论坛系统。路演时必然是由主讲人点进某一个特定的“重点演示节点”（如：微积分节点）。因此，**只为这个重点节点连接真实的 Supabase 数据库和评论接口**，地图上的其他所有子节点写死静态数据，点击无反应或弹出「节点建设中」。
   - **成效：** 至少省下3天数据结构设计与组件复用联调的时间。

3. **雷达匹配 (Matching) — 视联调进度降级为“动画播放器”：**
   - **优化：** 原计划前端与 Python 算法真实联调虽然很好，但在跨域、部署、请求网络延迟时常容易出玄学问题。如果在开发时碰壁超过 12 小时，应当**果断切断真实通信**。前端只负责播放 3 秒酷炫的雷达脉冲 CSS 动画，然后直接 `return` 预置的理想化 JSON 匹配结果，把节点用绿色高亮渲染出来。
   - **成效：** 保底方案，确保你的核心机制（雷达匹配效果）在演示时永不翻车。

## 四、 补充的最高优先级原则

1. **把 Next.js 视为共同战场：** 向团队明确，CS 和 EEE 都是前端开发，只是 CS 偏重 UI/交互，EEE 偏重数据流/接口对接。
2. **把“造假”提前：** 第一周周末，数据库里就必须躺着几十个假用户和帖子。没有数据，前端是无法进行排版和视觉测试的。
3. **牢记第一展示位：** 确保所有的开发时间都优先向 **React Flow 节点地图** 和 **Summon Oracle (H2A2H) 交互** 倾斜。哪怕点进子节点的论坛列表长得很简陋，只要入口处的节点图足够酷炫，AI 召唤的过程足够丝滑顺畅，路演的“Wow Moment (惊艳时刻)”就稳了。
